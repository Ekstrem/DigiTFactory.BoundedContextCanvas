# Bounded Context Canvas: BoundedContextCanvas

## 1. Имя

**BoundedContextCanvas** — ограниченный контекст управления описаниями архитектурных контекстов.

Техническое имя (namespace): `BoundedContextCanvas`
Команда-владелец: Platform Architecture Team

## 2. Назначение

Управление жизненным циклом описаний ограниченных контекстов: создание, редактирование, версионирование и публикация Bounded Context Canvas как стратегических артефактов организации.

## 7. Ubiquitous Language

Ключевые термины домена:

- **Bounded Context (Ограниченный контекст)** — автономная область бизнеса с чётко определёнными границами ответственности, языком и контрактами
- **Strategic Classification (Стратегическая классификация)** — категоризация контекста по важности (Core/Supporting/Generic), роли в бизнесе и уровню эволюции
- **Domain Role (Роль домена)** — архетип поведения контекста: Execution (исполнение), Gateway (интеграция), Analysis (аналитика)
- **Responsibility (Ответственность)** — обязательство контекста: что он делает и что явно НЕ делает
- **Public Interface (Публичный интерфейс)** — набор входящих команд/запросов и исходящих событий, через которые контекст взаимодействует с миром
- **Ubiquitous Language Term (Термин единого языка)** — ключевое понятие предметной области с определением, согласованным внутри контекста
- **External Connection (Внешняя связь)** — зависимость контекста от другого контекста с указанием паттерна интеграции (ACL, OHS, Conformist и т.д.)
- **Business Decision (Бизнес-решение)** — правило или политика, которую контекст обеспечивает и защищает
- **Assumption (Предположение)** — то, что считаем истинным при проектировании, но не доказано фактами
- **Verification Metric (Метрика верификации)** — измеримый показатель эффективности границ контекста
- **Canvas Version (Версия канваса)** — снимок состояния канваса в момент публикации; неизменяем после публикации

## 8. Бизнес-решения (Business Decisions)

1. **Strategic Alignment:** Core Domain не может иметь паттерн Conformist по отношению к Generic-контекстам. Если архитектор пытается установить такую связь — система отклоняет и предлагает ACL.

2. **Responsibility Cohesion:** Описание каждой ответственности должно быть уникальным в рамках канваса. Дублирование ответственностей сигнализирует о нечётких границах.

3. **Interface Integrity:** Каждое исходящее доменное событие должно быть обосновано хотя бы одной ответственностью. Нельзя публиковать событие, не связанное ни с одной обязанностью контекста.

4. **Naming Consistency:** Техническое имя (Namespace) должно соответствовать правилам именования компании (PascalCase, без спецсимволов, регулярное выражение `^[A-Z][a-zA-Z0-9]*$`).

5. **Contract Evolution Control:** При изменении PublicInterface (breaking change) — обязательное уведомление всех Downstream-потребителей с подтверждением совместимости.

## 9. Предположения (Assumptions)

1. Один агрегат `BoundedContext` достаточен для моделирования всех аспектов канваса (не требуется разделение на отдельные агрегаты для Language, Relationships и т.д.).
2. Канвас заполняется людьми (архитекторами), а не автоматически из кода — автоимпорт из CI/CD вторичен.
3. Количество контекстов в организации — десятки, не тысячи. Это определяет низкую нагрузку.
4. Связи между контекстами симметричны — если A связан с B, B знает о связи с A (через события).

## 10. Метрики верификации (Verification Metrics)

| Метрика | Источник | Целевое значение |
|---------|---------|-----------------|
| Процент контекстов с заполненным канвасом | Внутренний | > 80% всех продуктовых контекстов |
| Среднее количество ответственностей на контекст | Внутренний | 3–7 (слишком мало — слишком абстрактно, слишком много — God Context) |
| Процент событий без обоснования ответственностью | Внутренний | 0% |
| Время от создания Draft до Published | Внутренний | < 5 рабочих дней |
